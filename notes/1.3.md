# Chunk 1.3 — QC Checks

## What this is
The four-check quality gate that runs inside the Kafka consumer before any data reaches ClickHouse or S3/NetCDF.

## Where it lives in the pipeline
This is the core of Stage 3 — the only place in the entire pipeline where bad telemetry gets caught before poisoning the WeatherMesh forecast.

## The production constraint that matters
QC failures must never drop the packet or crash the consumer — flag it, pass it through marked, commit the offset. Silent drops = data loss. Crashes = lag builds = WeatherMesh misses its 10-minute window.

## What we built
pipeline/qc.py — a stateful QCChecker class running plausibility, relational (barometric formula), persistence (stuck sensor detection), and spatial-temporal (speed limit) checks. Wired into ingestion.py so the consumer calls checker.check(packet) on every message and prints PASS/FAIL with reasons. Fixed the producer to use realistic per-balloon position drift so spatial-temporal doesn't false-positive on random lat/lon jumps.

## What breaks if you get this wrong
Bad observations reach S3/NetCDF → WeatherMesh assimilates corrupted data → forecast error propagates to every customer. ClickHouse looking clean gives false confidence — WeatherMesh pulls from S3, not ClickHouse.

## Remember this
QCChecker must be instantiated once per consumer process, outside the message loop. Put it inside the loop and persistence/spatial-temporal state resets on every packet — those checks become useless.
