# Chunk 1.2 — Kafka Ingestion

## What this is
Kafka as the durable buffer between the HTTP ingestion endpoint and the Stage 3 consumer, with manual offset commit for at-least-once delivery.

## Where it lives in the pipeline
This is the handoff point between Stage 2 (ingestion) and Stage 3 — the HTTP endpoint produces to Kafka, and the consumer is where all real processing begins.

## The production constraint that matters
Consumer lag directly threatens the 10-minute WeatherMesh window. If the consumer falls behind, NetCDF writes are delayed, and the model assimilates stale data.

## What we built
`pipeline/ingestion.py` — a producer that sends 20 balloon packets partitioned by `balloon_id`, and a consumer with `enable_auto_commit=False` that commits only after processing. Includes sequence gap detection. Validated end-to-end using Kafka UI showing topic messages, consumer group lag draining to 0 after consume.

## What breaks if you get this wrong
Auto-commit means offset is committed before processing completes — crash between commit and ClickHouse write = data silently lost, no replay. Manual commit means crash before commit = replay on restart, handled by ReplacingMergeTree (ClickHouse) and S3 overwrite idempotency.

## Remember this
Lag = 0 after consume is the health signal. Lag growing = consumer behind = WeatherMesh window at risk.
